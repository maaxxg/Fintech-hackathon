# Task Brief ‚Äî Cashback Recommendation: End-to-End Integration

## Objective

Wire the cashback recommendation path end to end so that when the SvelteKit
frontend (Web branch) opens a high-priority client's detail page, it calls the
FastAPI backend (ML branch), which in turn calls `ModelToConfig.get_ranked_offers`
and returns a ranked list of personalised cashback offers. The UI then renders
them as "Required Actions".

Three concrete deliverables:

1. FastAPI `/clients/{client_id}/recommend` endpoint that returns a usable offer list.
2. Fix `ModelToConfig.py` so the dynamic cashback formula uses real
   `value_score` and churn `risk_score` (not the silent 0.5 fallback), and
   returns a clean, serialisable payload.
3. Frontend glue (on the Web branch) so `getRetentionMethods()` calls the real
   API and maps the response into `RetentionMethod[]`.

### Task B ‚Äî Backend: `/recommend` endpoint

Create `/Users/Max/Desktop/Fintech-hackathon/api/services/recommender.py`:

- `class RecommenderService` with:
  - `__init__`: loads `cashback_scores.parquet`, `features_value.parquet`,
    `churn_model.pkl`, and `features_churn.parquet` into memory once at
    startup. Builds lookup dicts keyed by `IDENTIFIKATOR_KLIJENTA`. Pre-computes
    churn probabilities for every client in `features_churn.parquet` using
    `churn_feature_columns.json` for the correct column order.
  - `recommend(client_id: str, top_n: int = 4) -> list[dict]`:
    1. Look up `value_score` (fallback `0.5` if client missing).
    2. Look up `risk_score` = churn probability (fallback `0.5` if missing).
    3. Call `ModelToConfig.get_ranked_offers(client_id, value_score=v, risk_score=r, top_n=top_n)`.
    4. Apply the minimal business rule from CLAUDE.md: keep only offers where
       `monthly_spend >= 0.5 * min_monthly_spend`. If nothing passes, return
       the top offer anyway (so the UI always has something to show during the
       demo ‚Äî document this deviation).
    5. For each offer, compute `expected_monthly_cashback =
       round(monthly_spend * percentage / 100, 2)` and add it to the dict.
    6. Add a `reason` string: top-1 ‚Üí `"Highest affinity (score {final_score:.2f})"`;
       others ‚Üí `"{category_share_spend*100:.0f}% spending share in {category}"`.
    7. Map `final_score` to a UI `priority`:
       - `>= 0.5` ‚Üí `"high"`
       - `>= 0.2` ‚Üí `"medium"`
       - else ‚Üí `"low"`

## Non-goals / Later

- Do **not** build the priority matrix, client list, or insights frontend
  pages here ‚Äî they live on the Web branch and are out of scope for this brief.
- Do **not** write to Firestore from the API. The endpoint is read-only and
  pure-compute.
- Do **not** implement ALS + LightGBM ranker from scratch. The existing
  `cashback_scores.parquet` already has the scored (client, category) table,
  and `ModelToConfig.py` is the business-logic layer on top of it. Reuse that.
- Do **not** persist priority scores. Priority fusion stays client-side in
  SvelteKit.
- No auth, no rate limiting, no TLS.

## Constraints / Caveats

- **Churn score source:** the simplest path is to run `churn_model.pkl` over
  `features_churn.parquet` once at service startup and cache a
  `client_id -> churn_prob` dict. Use `churn_feature_columns.json` to enforce
  column order. If that proves fiddly (dtype issues, missing columns for
  holdout), fall back to a hardcoded `0.5` churn score ‚Äî the value+scoring
  layer still works, just less personalised.
- **`cashback_scores.parquet` is not regenerated by this brief.** It already
  contains all holdout clients. If the file is missing, the endpoint will
  error ‚Äî that's acceptable; the pipeline is the source of truth for that file.
- **Category matching is fuzzy** (`_match_category` in `ModelToConfig.py`).
  Categories in the parquet contain mojibake (e.g. `OPSKRBA ELEKTRI√àNOM
  ENERGIJOM`), so the existing `latin1_fixes` dict in `_normalise` is
  load-bearing. Do not "clean" or rewrite that function.
- **Path to `cashback_scores.parquet`** is `config.PROCESSED_DATA_DIR /
  "cashback_scores.parquet"`. Project root is
  `/Users/Max/Desktop/Fintech-hackathon`. Start uvicorn from project root so
  relative paths resolve.
- **ID format:** `IDENTIFIKATOR_KLIJENTA` is a 12-char uppercase alphanumeric
  string (e.g. `BRJPQDRAHD40`). Case-sensitive. Do not lowercase.
- **CORS is required** ‚Äî without it, the browser blocks the cross-origin call
  from `localhost:5173` to `localhost:8000`.

## Acceptance criteria

- `uvicorn api.main:app --port 8000` starts without error, loading models
  and parquet files at startup.
- `curl http://localhost:8000/clients/BRJPQDRAHD40/recommend` returns a
  200 response whose `offers` array has at least one element with a non-null
  `percentage`, `expected_monthly_cashback`, and `priority`.
- Calling `/recommend` twice for the same client returns the same
  `percentage` (deterministic).
- `percentage` values visibly differ between a high-value/high-risk client and
  a low-value/low-risk client on the same category (proves the dynamic
  formula is wired correctly, not stuck at 0.5/0.5).
- `scripts/smoke_recommend.py` runs cleanly and prints offers for 3 holdout
  clients.
- On the Web branch, opening a client detail page against a locally running
  API renders the offers in the "Required Actions" panel with the real
  percentages and descriptions (no fallback mock used).
